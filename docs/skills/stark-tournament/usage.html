/stark-tournament

Run multi-LLM competitions — N competitors, 1 task, 1 winner
USAGE GUIDE
Quick Start
# Simplest: inline prompt, default competitors (claude, codex, gemini)
/stark-tournament "Write a Python function to parse ISO 8601 dates"

# Pick competitors and evaluation strategy
/stark-tournament "Implement a Redis cache wrapper" --competitors claude,codex --strategy test

# Full config file for complex tournaments
/stark-tournament --config tournament.yaml

Evaluation Strategies

semantic

Judge reads all outputs and scores them on configurable factors (correctness, completeness, quality). Best for prose, code design, and general-purpose comparisons. Default strategy.

visual

Judge compares visual/rendered outputs. Useful for HTML, CSS, UI component generation, or any task where appearance matters alongside correctness.

test

Outputs are evaluated by running a test suite (--test-file required). Scores based on pass/fail counts. Objective and repeatable — ideal for algorithmic tasks.

Arguments Reference
Argument Type Default Description
"prompt" positional — Inline prompt text (quoted). Mutually exclusive with --config.
--config path — YAML config file for complex tournament setups.
--strategy enum semantic Evaluation strategy: semantic visual test
--competitors list claude,codex,gemini Comma-separated competitor IDs.
--factors key=weight correctness=2.0 completeness=1.0 quality=1.0 Evaluation factors with weights.
--judge model claude-sonnet-4-6 Model used to judge competitor outputs.
--test-file path — Test file for test strategy (required when --strategy test).
--output-dir path auto Directory for output files.
--timeout seconds 300 Timeout per competitor.
--variables key=value — Template substitution variables for prompt.
--keep-all flag off Retain all competitor outputs, not just winner.
--json flag off Output structured JSON instead of human-readable table.
--dry-run flag off Print config and exit without running.

Common Workflows

Quick Comparison

Compare all 3 LLMs on a coding task with defaults. Just provide the prompt — everything else is automatic.

/stark-tournament "Write a retry decorator with exponential backoff"

Test-Driven Tournament

Evaluate outputs by running an existing test suite. Objective scoring — no judge subjectivity.

/stark-tournament "Implement merge sort" --strategy test --test-file tests/test_sort.py

Custom Factors

Weight what matters most. Emphasize security for auth code, performance for hot paths, readability for shared libs.

/stark-tournament "JWT auth middleware" --factors security=3.0 correctness=2.0 readability=1.0

Head-to-Head

Pit two specific models against each other when you have a strong hunch but want data.

/stark-tournament "GraphQL resolver" --competitors claude,gemini

YAML Config Tournament

For repeatable, complex setups — multiple variables, custom prompts, specific output dirs.

/stark-tournament --config .tournaments/api-design.yaml --keep-all

Dry Run Preview

Validate your setup before burning API credits. See exactly what will run.

/stark-tournament "Complex task" --competitors claude,codex,gemini --dry-run
Failure Recovery
Failure What Happens
No prompt or config Asks you: "What should the competitors work on?"
Config file not found Error with path shown. Check the --config path.
Single competitor fails Disqualified — tournament continues with remaining competitors.
All competitors fail Error suggesting to check --timeout and --competitors values.
Judge fails Fallback: first valid output used, eval_failed: true in audit.
Timeout exceeded Timed-out competitor disqualified, others continue.
